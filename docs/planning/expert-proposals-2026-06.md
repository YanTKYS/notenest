## 採用判断の軸

| 軸 | 判断内容 |
|----|----------|
| ローカル動作 | 閉域・ローカルで動くか。外部通信・クラウド・AI API 前提でないか |
| 保存形式 | `.notenest` / `.ideanest` / `.chatnest` / TempNest JSON の形式変更を伴わないか |
| 日常操作改善 | 日常のメモ・整理・記録操作の手数を減らせるか |
| Workspace の軽さ | 既存 Workspace のシンプルさ・独立性を壊さないか |
| 新ユーザー負荷 | 新規ユーザーの混乱を増やさないか |

## C. 長期構想・保留

**方針:** 魅力はあるが、保存形式変更・索引・UX 設計が大きくなるため短期実装しない。将来の可能性を残すため「対象外」ではなく「保留」として整理。

- 共通データモデルへの全面移行
- `.nestsuite` 統合ファイル形式
- クロス Workspace 双方向リンク
- クロス Workspace 全文検索
- ナレッジグラフ
- Day / Week Review（AI なしローカル集計版でも設計が大きい）
- セッション復元の選択的復元
- 複数 Window レイアウト保存
- テンプレート機能
- パフォーマンス自己診断
- ErrorLogService ローテーション

**整理メモ:**

- クロス Workspace 検索や双方向リンクは有用だが、保存形式・索引・UX 設計が大きくなるため短期実装しない
- `.nestsuite` 統合ファイル形式は既存 `.notenest` / `.ideanest` / `.chatnest` 方針（疎連携・独立保存）と衝突するため構想扱いに留める
- Day / Week Review は AI なしローカル集計なら将来可能だが、まずコピー・投入・すべて保存など低リスク改善を優先する

---

## D. 当面対象外

**方針:** NestSuite 本体の方針（ローカルファースト・閉域・外部通信なし・軽量・単体 EXE）と合わないもの。「採用しない」ではなく「現時点では対象外」として整理する。

- 外部 AI API 前提の要約・タグ付け
- クラウド同期
- CRDT / 共同編集
- プラグイン / スクリプト実行基盤
- 高機能リッチエディタ全面移行
- 画像 / リッチメディア管理
- 常時バックグラウンド同期

**理由:**

| 提案 | 対象外の理由 |
|------|------------|
| 外部 AI 前提の機能 | 閉域利用・外部通信なしの方針と衝突する |
| クラウド同期 | ローカルファースト方針と衝突。OneDrive 等の手動配置で代替可能 |
| 共同編集 / CRDT | 排他制御・マージ処理が複雑。ローカル単一ファイル管理方針と根本的に合わない |
| プラグイン基盤 | セキュリティ評価・運用負荷が大きい。別プロダクト化しやすい |
| 高機能リッチエディタ | エディタ部品の差し替えは影響範囲が大きい。安定性優先のため見送り |
| 画像 / リッチメディア | 単一 JSON への画像埋め込みはファイルサイズ増大を招く |
